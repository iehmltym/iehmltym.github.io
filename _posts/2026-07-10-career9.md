# 保守支援 AI Agent PoC 面试问答库

## 0. 项目一句话版本

### Q. このプロジェクトを簡単に説明してください。

大手事務機器メーカー向けに、プリンター・スキャナーおよび関連クラウドサービスの製品マニュアル、保守FAQ、問い合わせ履歴を活用して、保守担当者やサポート担当者の問い合わせ対応を支援する AI Agent 型チャット機能を開発しました。

私は開発メンバーとして、Python / LangChain を用いた RAG 処理、問い合わせ内容に応じた簡易 Tool Calling、Gemini API による回答生成、ストリーミング応答、会話履歴管理、PoC評価用ログ出力を担当しました。

## 1. 自己紹介

### Q. 自己紹介をお願いします。

これまで業務システム開発を中心に経験してきました。直近では、メーカー業界向けの業務改善 AI Agent PoC 開発に参画しています。

このプロジェクトでは、事務機器メーカーのプリンター、スキャナー、関連クラウドサービスに関する製品マニュアル、保守FAQ、問い合わせ履歴を活用し、保守担当者やサポート担当者の問い合わせ一次回答、製品仕様確認、障害切り分けを支援する AI Agent 型チャット機能を開発しています。

私は主に、Python / LangChain を用いた RAG 処理、Embedding、Vector Search、Retriever、Prompt Template、問い合わせ内容に応じた簡易 Tool Calling、Gemini API 連携、ストリーミング応答、会話履歴管理、ログ出力を担当しました。

PoC なので、単に回答できるだけでなく、質問内容、検索結果、回答内容、Token 使用量、レスポンス時間、エラー情報をログに残し、回答品質や処理時間を評価できるようにした点を意識しました。

### Q. 30秒で説明してください。

直近では、メーカー業界向けの保守支援 AI Agent PoC に参画しています。製品マニュアル、保守FAQ、問い合わせ履歴を対象に、Python / LangChain で RAG 処理を実装し、問い合わせ内容に応じて検索先を切り替える簡易 Tool Calling を担当しました。Gemini API による回答生成、ストリーミング応答、会話履歴管理、ログ出力も実装し、PoC評価に使える形にしました。

## 2. 業務背景

### Q. なぜこのAI Agentが必要だったのですか？

保守担当者やカスタマーサポート担当者が問い合わせ対応を行う際、製品マニュアル、FAQ、過去の問い合わせ履歴を個別に確認する必要がありました。そのため、調査時間が長くなること、担当者によって回答品質にばらつきが出ること、ナレッジが属人化することが課題でした。

このPoCでは、問い合わせ内容に応じて必要な情報を検索し、回答根拠を提示しながら回答を生成することで、問い合わせ一次回答や障害切り分けを支援することを目的にしました。

### Q. 実際の利用者は誰ですか？

主な利用者は、保守担当者、カスタマーサポート担当者、社内ヘルプデスク担当者を想定していました。一般ユーザーに直接公開するものではなく、社内担当者が問い合わせ対応を行う際の調査支援ツールという位置づけです。

### Q. どんな業務に使う想定ですか？

主に以下のような業務を想定していました。

- 操作方法の確認
- 製品仕様の確認
- 障害切り分け
- よくある問い合わせの確認
- 過去の類似事例の確認
- 回答根拠の確認

### Q. なぜ事務機器メーカーとAI Agentは相性が良いのですか？

事務機器メーカーでは、プリンターやスキャナー、クラウドサービスに関する製品マニュアル、保守FAQ、問い合わせ履歴、設定手順書など、文書ベースのナレッジが多いです。問い合わせ対応でも、それらの資料を参照して回答するケースが多いため、RAG や AI Agent と相性が良いと考えました。

## 3. 担当範囲

### Q. あなたの担当範囲はどこまでですか？

私は開発メンバーとして、主に AI 機能とバックエンド実装を担当しました。具体的には、RAG処理、簡易 Tool Calling、Gemini API 連携、ストリーミング応答、会話履歴管理、ログ出力、単体テスト、API動作確認、不具合修正です。

要件定義や顧客折衝は主にリーダーや上位SEが担当していました。私は設計メモやチケットをもとに実装を進めました。

### Q. 設計は担当しましたか？

基本設計全体は上位SEが担当していました。私は詳細設計に近い部分として、RAG処理の流れ、APIの入出力、ログ出力項目、エラーハンドリング、検索対象の切り替え処理など、実装に近い設計を担当しました。

### Q. 顧客と直接やり取りしましたか？

顧客との直接折衝は主にリーダーや上位SEが担当していました。私は開発メンバーとして、Redmine / JIRA のチケット、設計メモ、検証観点をもとに実装し、不明点がある場合はチャットや定例で確認していました。

### Q. 日语沟通怎么处理？

基本的には、チケット、設計メモ、チャット、定例会で確認しながら進めました。複雑な仕様確認は、できるだけ文章で整理して確認するようにしていました。口頭だけで曖昧に進めず、実装内容や確認事項をチケットやメモに残すことを意識しました。

## 4. RAG

### Q. RAGの処理の流れを説明してください。

まず、製品マニュアル、保守FAQ、問い合わせ履歴をチャンク単位に分割します。次に、文書種別、製品名、カテゴリ、ページ番号などのメタデータを付与し、Embedding 化して Vector DB に保存します。

ユーザーから質問が来たら、質問文も Embedding 化して類似検索を行い、関連文書を取得します。その検索結果を Prompt に含めて LLM に渡し、回答を生成します。

### Q. なぜRAGを使うのですか？

LLM だけでは社内の製品マニュアルや保守FAQ、問い合わせ履歴のような個別業務データを知りません。RAG を使うことで、外部の業務文書を検索し、その内容をもとに回答を生成できます。また、回答に参照元を含めることで、担当者が根拠を確認しやすくなります。

### Q. chunkingで気をつけたことは？

チャンクが大きすぎると不要な情報が混ざり、小さすぎると文脈が不足します。そのため、マニュアルでは見出しや手順単位、FAQでは質問と回答の単位、問い合わせ履歴では1件の問い合わせ単位を意識して分割しました。

### Q. metadataには何を入れましたか？

文書ID、文書種別、製品名、カテゴリ、タイトル、ページ番号、更新日などを想定しました。特に、回答時に参照元を表示するため、文書名やページ情報、カテゴリは重要でした。

### Q. Vector Search は何のために使いましたか？

問い合わせ文と意味的に近いマニュアルやFAQを検索するためです。単純なキーワード検索だけだと、表現が少し違う場合に検索できないことがあります。Embedding を使ってベクトル化することで、意味的に近い文書を取得しやすくなります。

### Q. 検索精度が悪い場合はどうしましたか？

主に以下を調整しました。

- chunk サイズ
- metadata
- top-k
- Prompt
- 検索対象の切り替え条件
- テストデータ

特に、FAQ、マニュアル、問い合わせ履歴では文書の粒度が違うため、同じ前処理ではなく、文書種別ごとに調整することを意識しました。

## 5. Agent / Tool Calling

### Q. Agent と RAG の違いは何ですか？

RAG は、質問に関連する文書を検索し、その検索結果を LLM に渡して回答を生成する仕組みです。一方、Agent は、質問内容に応じてどのツールや処理を使うかを判断する仕組みです。

今回のPoCでは、操作方法や仕様確認であれば製品マニュアル検索、よくある問い合わせであれば保守FAQ検索、類似事例確認であれば問い合わせ履歴参照を使うように、検索対象を切り替える簡易 Tool Calling を実装しました。

### Q. Tool Calling では何を切り替えましたか？

主に以下の3種類です。

- 操作方法・仕様確認：製品マニュアル検索
- よくある問い合わせ：保守FAQ検索
- 類似事例確認：問い合わせ履歴参照

すべてを同じ検索対象にすると関係ない文書が混ざりやすいため、問い合わせ内容に応じて参照先を分けました。

### Q. なぜ複雑なAgentにしなかったのですか？

PoC段階では、まず検索精度、回答品質、レスポンス時間、ログ取得を検証することが重要だったためです。複雑な自律型Agentにすると処理の追跡や評価が難しくなります。そのため、最初は検索対象を限定し、ログで動作を確認しやすい構成にしました。

### Q. LangChainのAgent Executorを使いましたか？

本格的な複雑な Agent Executor というより、PoC として限定的な Tool Calling 構成にしました。LangChain では主に Retriever、Prompt Template、Chain 構成を使い、検索対象の切り替え部分はシンプルに制御しました。正式導入を想定する場合は、Agent Executor や LangGraph のような構成も検討対象になると思います。

### Q. Tool Calling の失敗時はどうしましたか？

検索対象が判断できない場合は、デフォルトで製品マニュアルとFAQを優先して検索する方針にしました。また、検索結果が少ない場合は回答を断定せず、関連資料が不足している旨を返すように Prompt で制御しました。

## 6. LangChain

### Q. LangChainでは何を使いましたか？

主に Retriever、Prompt Template、Chain 構成を使いました。文書をチャンク分割して Embedding 化し、Vector Search で関連文書を取得し、その結果を Prompt Template に組み込んで LLM に渡す流れです。

### Q. なぜLangChainを使いましたか？

RAG や Tool Calling のような LLM アプリケーションで必要になる部品が整理されているためです。Retriever、Prompt Template、Chain などを使うことで、処理の流れを分けて実装しやすくなります。また、PoC段階で構成を変更しながら検証しやすい点もメリットでした。

### Q. LangChainで苦労した点は？

文書の前処理や検索結果の品質調整です。LangChain自体を使うだけでは精度は上がらないため、chunking、metadata、Prompt、検索条件を調整する必要がありました。

## 7. LLM API

### Q. Gemini API と OpenAI API の使い分けは？

主に Gemini API を使って回答生成処理を実装しました。OpenAI API は比較検証用として確認しました。PoCでは、回答品質、レスポンス時間、コスト、運用面を比較できるようにする目的でした。

### Q. temperatureなどのパラメータは調整しましたか？

はい。業務支援用途では回答の安定性が重要なので、創造性よりも安定性を重視して低めの temperature を想定しました。回答がぶれすぎないようにし、参照文書に基づいた回答になるよう Prompt と合わせて調整しました。

### Q. hallucination対策は？

検索結果に基づいて回答するよう Prompt を調整しました。また、回答に参照元文書を含めることで、担当者が根拠を確認できるようにしました。検索結果に十分な情報がない場合は、断定的に回答しない方針にしました。

### Q. ストリーミング応答はなぜ使いましたか？

LLM の回答生成には時間がかかる場合があります。ストリーミング応答を使うことで、回答全体が完了する前に生成されたテキストを順次表示でき、ユーザーの待ち時間を短く感じさせることができます。

## 8. PoC評価

### Q. PoCでは何を評価しましたか？

主に以下を評価しました。

- 検索結果の妥当性
- 回答品質
- 参照元の正確性
- レスポンス時間
- Token使用量
- エラー発生状況
- ツール選択の妥当性

### Q. ログには何を出しましたか？

質問内容、選択された検索処理、検索結果、回答内容、Token使用量、レスポンス時間、エラー情報を出力しました。PoC後に、どの質問で検索がうまくいったか、どの回答に改善が必要かを確認するためです。

### Q. 成果はどう説明しますか？

正式導入前のPoCなので、定量的な業務削減効果までは出していません。ただし、保守FAQ、製品マニュアル、問い合わせ履歴を切り替えて参照し、回答根拠付きで回答する AI Agent 型チャット機能を実装しました。また、評価用ログを残すことで、今後の精度改善や正式導入判断に使える状態にしました。

### Q. PoCと本番開発の違いは何ですか？

PoCでは、まず実現可能性や回答品質、検索精度、レスポンス時間を確認することが目的です。本番開発では、認証認可、権限管理、監査ログ、可用性、運用監視、データ更新フロー、セキュリティなどをより厳密に設計する必要があります。

## 9. API / Backend

### Q. APIでは何を実装しましたか？

チャットAPI、会話履歴API、ログ出力APIを実装しました。チャットAPIでは、質問を受け取り、検索対象の判定、RAG検索、LLM回答生成、ストリーミング応答までを処理します。ログ出力APIでは、PoC評価に必要な情報を確認できるようにしました。

### Q. エラーハンドリングはどうしましたか？

LLM APIエラー、検索結果なし、タイムアウト、入力不正などを想定しました。APIエラー時にはリトライやフォールバックを行い、検索結果がない場合は無理に回答せず、情報不足であることを返す方針にしました。

### Q. 会話履歴はなぜ必要ですか？

問い合わせ対応では、前の質問の続きとして聞かれることがあります。セッション単位で会話履歴を持つことで、直前の文脈を反映した回答ができるようになります。ただし、履歴を長くしすぎると token が増えるため、保持する範囲は制限しました。

## 10. Security / 運用

### Q. セキュリティ面で気をつけたことは？

PoC段階では簡易的な確認が中心でしたが、API Keyをソースに直接書かないこと、ログに不要な個人情報を残さないこと、アップロードファイルや問い合わせ履歴の取り扱いに注意することを意識しました。本番導入では、権限管理、監査ログ、データマスキングが必要になると考えています。

### Q. 社内文書をLLMに渡すリスクは？

社内文書には機密情報が含まれる可能性があります。そのため、どのデータを外部APIに渡すか、ログに何を残すか、権限ごとに検索範囲を制限するかが重要です。本番導入では、社内ルールやセキュリティポリシーに従った設計が必要です。

### Q. 本番化するなら何を追加しますか？

認証認可、文書ごとのアクセス制御、監査ログ、管理画面、データ更新バッチ、精度評価ダッシュボード、障害監視、コスト監視を追加したいです。また、ユーザーフィードバックを収集し、検索条件やPrompt改善に活用できる仕組みも必要です。

## 11. 弱点を突かれた時の答え

### Q. OCRは担当しましたか？

本格的なOCRモデルの開発は担当していません。私の担当は、テキスト化された製品マニュアルやFAQ、問い合わせ履歴をRAGやAgentの検索対象として扱う部分です。文書データのチャンク分割、メタデータ付与、Embedding、検索、LLM連携が中心です。

### Q. 顧客折衝はできますか？

これまで顧客折衝を主担当した経験は多くありません。現場では、リーダーや上位SEが顧客対応を担当し、私は開発メンバーとして実装や技術調査を担当することが多かったです。ただし、仕様確認や進捗共有については、チケットやチャット、定例会で対応していました。

### Q. Pythonはどの程度できますか？

Pythonは、主に生成AI PoCやデータ処理、LangChainを使ったRAG処理で使用しました。Javaのように長い業務システム経験があるわけではありませんが、API連携、文書処理、Embedding、LangChainを使った実装には対応できます。

### Q. AIモデル自体を作ったのですか？

AIモデル自体をゼロから学習したわけではありません。既存のLLM APIを利用し、業務データや社内文書と連携させるアプリケーション側の実装を担当しました。具体的には、RAG、Tool Calling、Prompt、ログ出力、API連携です。

### Q. 機械学習の専門家ですか？

機械学習研究者というより、業務システム開発経験をベースに、生成AIを業務アプリケーションに組み込む実装寄りのエンジニアです。LLMやRAGを使って、既存業務データを活用する仕組みを作ることに強みがあります。

## 14. 実戦回答原稿

この章は、面接でそのまま話しやすい原稿形式です。上の章は質問別の整理、この章は実際に口頭で説明する時の流れとして使います。

### 自己紹介 長め版

私はこれまで、主に業務システム開発を中心に経験してきました。直近では、メーカー業界向けの業務改善AI Agent PoC開発に参画しています。

このプロジェクトは、事務機器メーカー向けの保守支援・問い合わせ対応を効率化するためのPoCです。対象は、プリンターやスキャナー、関連クラウドサービスに関する製品マニュアル、保守FAQ、過去の問い合わせ履歴です。

従来は、サポート担当者や保守担当者が問い合わせを受けた際に、複数のマニュアルやFAQ、過去事例を手動で確認していたため、調査に時間がかかることや、担当者によって回答品質にばらつきが出ることが課題でした。

そこで、問い合わせ内容に応じて、製品マニュアル検索、保守FAQ検索、問い合わせ履歴参照を切り替え、RAGとLLMを使って回答を生成するAI Agent型チャット機能を開発しています。

## 15. 実戦Q&A 追加版

### Q1. Agent と RAG の違いは何ですか？

RAGは、ユーザーの質問に対して関連文書を検索し、その検索結果をLLMに渡して回答を生成する仕組みです。一方でAgentは、質問内容に応じてどの処理やツールを使うかを判断する構成です。

今回のPoCでは、RAGだけではなく、問い合わせ内容に応じて製品マニュアル検索、保守FAQ検索、問い合わせ履歴参照を切り替える簡易的なTool Callingを実装しました。そのため、複雑な自律型Agentではありませんが、業務用途に合わせたAI Agent型の構成になっています。

### Q2. なぜTool Callingが必要だったのですか？

問い合わせ内容によって参照すべきデータが違うためです。例えば、操作方法や製品仕様に関する質問であれば製品マニュアルを参照します。よくある問い合わせであれば保守FAQ、過去の類似事例を確認したい場合は問い合わせ履歴を参照します。

すべてを同じ検索対象にすると、関係ない文書が混ざりやすくなるため、問い合わせ種別に応じて検索先を分ける構成にしました。

### Q3. LangChainでは何を使いましたか？

主にRetriever、Prompt Template、Chain構成を使いました。文書をチャンク分割してEmbedding化し、Vector Searchで関連文書を取得します。取得した文書をPrompt Templateに組み込み、LLMに渡して回答を生成する流れです。

### Q4. Vector Search は何のために使いましたか？

ユーザーの問い合わせ文と意味的に近いマニュアルやFAQを検索するために使いました。単純なキーワード検索だけだと、表現が少し違う場合に検索できないことがあります。Embeddingを使ってベクトル化し、意味的に近い文書を取得することで、より関連性の高い情報をLLMに渡せるようにしました。

### Q5. 回答の正確性はどう確認しましたか？

PoCでは、質問内容、検索結果、回答内容、参照元文書、Token使用量、レスポンス時間をログに出力しました。そのログをもとに、期待した文書が検索されているか、回答が参照元に基づいているかを確認しました。

また、回答精度確認用のテストデータを作成し、プロンプトや検索条件を調整しました。

### Q6. OpenAI API と Gemini API の両方を使ったのですか？

主にGemini APIを使って回答生成処理を実装しました。OpenAI APIについては、モデル切り替えや比較検証用として確認しました。PoC段階だったため、回答品質やレスポンス、コスト面を比較できるようにする目的です。

### Q7. OCR や画像認識も担当しましたか？

本格的なOCRモデルの開発は担当していません。私の担当は、テキスト化された製品マニュアルやFAQ、問い合わせ履歴をRAGやAgentの検索対象として扱う部分です。文書データのチャンク分割、メタデータ付与、Embedding、検索、LLM連携が中心です。

### Q8. あなたの担当範囲はどこまでですか？

要件定義や顧客折衝は主にリーダーや上位SEが担当していました。私は開発メンバーとして、設計メモやチケットをもとに、RAG処理、Tool Calling、LLM API連携、ログ出力、単体テスト、API動作確認、不具合修正を担当しました。

### Q9. 一番工夫した点は何ですか？

検索対象を分けた点です。製品マニュアル、保守FAQ、問い合わせ履歴は文書の粒度や使い方が違うため、同じ検索処理にまとめると精度が落ちやすいと考えました。そのため、問い合わせ内容に応じて検索対象を切り替える構成にし、回答には参照元情報も含めるようにしました。

### Q10. 苦労した点は何ですか？

検索結果の品質調整です。チャンクが大きすぎると不要な情報が混ざり、小さすぎると文脈が不足します。そのため、チャンク分割の単位やメタデータ、検索条件、プロンプトを調整しながら、回答に必要な情報が取得できるようにしました。

### Q11. なぜこのAI Agentを作る必要があったのですか？

保守担当者やサポート担当者が問い合わせ対応時に、製品マニュアル、FAQ、過去の問い合わせ履歴を個別に確認しており、調査時間がかかっていたためです。また、担当者によって回答品質にばらつきが出ることも課題でした。

そのため、問い合わせ内容に応じて必要な情報を検索し、回答根拠を含めて回答できるAI Agent型チャットをPoCとして開発しました。

### Q12. 実際の利用者は誰ですか？

主な想定利用者は、保守担当者、カスタマーサポート担当者、社内ヘルプデスク担当者です。エンドユーザー向けというより、社内担当者の問い合わせ一次回答や調査作業を支援する用途です。

### Q13. どんな問い合わせを想定していましたか？

操作方法の確認、製品仕様の確認、よくある障害の切り分け、過去の類似問い合わせの確認などです。例えば、特定のエラー表示が出た場合の確認手順や、クラウド連携機能の設定方法などを想定していました。

### Q14. 設計は担当しましたか？

基本設計全体は上位SEが担当していました。私は詳細設計レベルで、RAG処理の流れ、ログ出力項目、APIの入出力、エラーハンドリングなど、実装に近い部分を担当しました。

### Q15. チーム内でどうやって作業していましたか？

RedmineやJIRAのチケット、設計メモをもとに作業していました。不明点がある場合は、チャットや定例でリーダーに確認しながら進めました。実装後は単体テスト、API動作確認、不具合修正まで担当しました。

### Q16. RAGの処理の流れを説明してください。

まず、製品マニュアルやFAQ、問い合わせ履歴をチャンク単位に分割します。その後、メタデータを付与してEmbedding化し、Vector DBに保存します。

ユーザーから質問が来たら、質問文もEmbedding化して類似検索を行い、関連文書を取得します。取得した文書をPromptに含めてLLMに渡し、回答を生成します。

### Q17. chunkingで気をつけたことは？

チャンクが大きすぎると不要な情報が混ざり、小さすぎると文脈が不足します。そのため、マニュアルやFAQの文書構造を見ながら、見出しや項目単位で分割することを意識しました。また、製品名、カテゴリ、文書種別などのメタデータも付与しました。

### Q18. metadataには何を入れましたか？

文書ID、文書種別、製品名、カテゴリ、タイトル、ページ番号、更新日などを想定していました。回答時に参照元を表示できるようにするため、文書名やページ情報は特に意識しました。

### Q19. どのように検索対象を切り替えましたか？

問い合わせ内容をもとに、操作方法や仕様確認であれば製品マニュアル検索、よくある問い合わせであれば保守FAQ検索、過去事例確認であれば問い合わせ履歴参照を使う構成にしました。

PoC段階だったため、まずは限定的なルールとプロンプトで検索先を切り替える簡易Tool Callingとして実装しました。

### Q20. LangChainではAgent Executorを使いましたか？

本格的な複雑なAgent Executorというより、PoCとして限定的なTool Calling構成にしました。LangChainでは主にRetriever、Prompt Template、Chainの構成を使い、検索対象の切り替え部分はシンプルに制御しました。

正式導入を想定する場合は、Agent ExecutorやLangGraphのような構成も検討対象になると思います。

### Q21. なぜ複雑なAgentにしなかったのですか？

PoC段階では、まず回答品質や検索精度を確認することが重要だったためです。複雑なAgentにすると、処理の追跡や評価が難しくなるため、最初は検索対象を限定し、ログで動作を確認しやすい構成にしました。

### Q22. Gemini APIとOpenAI APIの使い分けは？

主にGemini APIを使って回答生成処理を実装しました。OpenAI APIは比較検証用として確認しました。PoCでは、回答品質、レスポンス時間、コスト、運用面を比較できるようにする目的でした。

### Q23. hallucination対策はしましたか？

検索結果に基づいて回答するようにPromptを調整しました。また、回答に参照元文書を含めることで、担当者が根拠を確認できるようにしました。検索結果に十分な情報がない場合は、断定的に回答しないようにする方針にしました。

### Q24. ストリーミング応答はなぜ使いましたか？

LLMの回答生成には時間がかかることがあるため、ユーザーの待ち時間を短く感じてもらうためです。回答全体が完了する前に、生成されたテキストを順次表示することで、操作感を改善しました。

### Q25. PoCでは何を評価しましたか？

主に、検索結果の妥当性、回答品質、参照元の正確性、レスポンス時間、Token使用量、エラー発生状況を確認しました。正式導入前に、業務で使えるレベルかどうかを判断するためです。

### Q26. ログには何を出しましたか？

質問内容、選択された検索処理、検索結果、回答内容、Token使用量、レスポンス時間、エラー情報を出力しました。PoC後に、どの質問で検索がうまくいったか、どの回答に改善が必要かを確認するためです。

### Q27. 成果はどのように説明できますか？

正式導入前のPoCなので、定量的な業務削減効果までは出していません。ただし、保守FAQ、製品マニュアル、問い合わせ履歴を横断的に参照し、回答根拠付きで回答する仕組みを実装できました。

また、評価用ログを残すことで、今後の精度改善や正式導入判断に使える状態にしました。
